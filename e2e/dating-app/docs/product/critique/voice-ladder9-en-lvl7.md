## Top 3: fix now

1. **[blocker][product] PRD-EVENTS-E1-S3:18**, this story sells concert tickets inside a dating app: city line-ups, four tickets an order, ten-minute seat holds, QR e-tickets, genre filters. The BRD hired exactly three buyers, all to find a sustained real connection (VISION:51), and the only goals on the board are MAU, weekly-match-rate, and premium break-even (BRD-G3:56). Whoever owns this spec started building a second company inside the first and did not notice. A concert-ticketing engine serves none of those jobs, and the file's own gold-plating note admits it. That is an admission, not a defense. → **Blown apart because:** a national ticketing platform eats the design, build, and ops capacity the unproven north-star (match to sustained chat) needs, moves no BRD metric, has no buyer in the persona set, and drags the team into event logistics they have no competence in. You ship the thing nobody hired you for while the core loop sits half-done. → **Rewrite it properly:** cut PRD-EVENTS-E1-S3 from the spec entirely. If the offline-community vision matters, park it as a one-line idea in the vision doc, not a fully-specced story with AC that invites estimation. `DEC-worthy` (binding scope ruling: a `scope:out`/`could`/`later` PRD that contradicts the build focus).

2. **[blocker][tech, product] PRD-CHAT-E1-S4:18**, both acceptance criteria are unfalsifiable adjectives: the conversation must be "absolutely safe" and the system must "fully protect" the user. No actor, no attack surface, no observable outcome. This is the safety story, in a trust-critical product. Reading an AC that says "absolutely safe" in a safety story is where I stopped trusting the rest of the spec. → **Blown apart because:** no engineer can write a test for "absolutely safe", so "given bad behavior, then system fully protects" passes for any implementation including doing nothing. The story either blocks the chat epic forever or gets waved through as theatre, and it duplicates the real trust scope already in PRD-SAFETY. → **Rewrite it properly:** delete the story and route any chat-specific safety need into PRD-SAFETY as concrete behaviour, e.g. "Given a message contains a phone-number pattern, when delivered, then a safety-tip overlay shows, the message is not blocked, and the event is logged for moderation"; "Given a user taps Block, then no further messages are delivered either way and the blocked user cannot start a new conversation."

3. **[blocker][market] BRD:58**, BRD-G3 promises revenue break-even by Year 2, yet PRD-PREMIUM is `horizon:later` and Year 1 builds 100k MAU, identity-verification infra, and a realtime messaging backend with zero premium revenue. There is no target ARPU, no Year 1 conversion rate, no cost-to-serve estimate anywhere. Industry benchmark is 5-10% MAU-to-paid, so 5k-10k payers at 100k MAU, and the premium set is self-confessed me-too (PRD-PREMIUM:42). The spec noticed the chicken-and-egg problem and then just wrote the number down anyway. The break-even claim is asserted, not modeled. → **Blown apart because:** an app deferring all revenue to Year 2 while carrying Year 1 infra and moderation costs needs a funding bridge the spec never names, so the team either cuts safety/chat corners to lower burn or simply misses break-even, discovered too late to course-correct. → **Rewrite it properly:** add a unit-economics section to brd.md (target ARPU in VND, end-of-Year-1 conversion %, cost-to-serve per MAU at 100k); if Year 1 is pre-revenue by design, say so and add a runway/funding constraint to Constraints. `DEC-worthy` (binding goal-vs-horizon ruling on BRD-G3). ### Tech

- **[blocker] PRD-CHAT-E1-S4:18**: AC "absolutely safe" / "fully protect" are non-observable; no test can be written, see Top 3 #2. → **Blown apart because:** the AC never falsifies, so the story ships with security holes invisible. → **Rewrite it properly:** specific falsifiable controls (phone-pattern overlay + log; Block stops delivery both ways) naming mechanism, observable outcome, and boundary.

- **[blocker] PRD-CHAT-E1-S3:18**: "smooth" / "fast and not laggy" with no numeric bound, device class, OS, list size, or frame-rate floor. Any implementation passes; 60fps on a Pixel 9 and 12fps on a low-end device both satisfy "smooth". Not-Testable per INVEST, and the `should` label makes it likely to ship unverified. → **Blown apart because:** no threshold means no deterministic evaluation. → **Rewrite it properly:** anchor to the target device class, "100 messages, continuous scroll, at least 60fps on a mid-range Android (Snapdragon 665-class)"; "open conversation, 50 messages load, first message within 1.5s on 4G".

- **[major] PRD-MATCH-E1-S2:21**: the AC embeds a weeks-long live experiment, swipe-right rate at least 20% over random control, A/B with p<0.05, as the acceptance criterion of a now/must story. It cannot be evaluated at story close; it needs accumulated live traffic and an A/B infra the spec never defines. → **Blown apart because:** the story can only be "shipped to be evaluated later", not "done", breaking the Independent and Testable axes of INVEST and hiding quality debt. → **Rewrite it properly:** split into two. Keep the build story with deterministic AC ("offline on 10k historical swipe events, precision@5 at least distance-only baseline +10%"), and a separate experiment epic owning the A/B design, infra, significance tracking, and rollout decision.

- **[major] PRD-AIREC-E1-S1:20**: the AC references "a match converting to sustained messaging" as a training signal, produced by PRD-CHAT, but no event schema, API contract, or event-bus spec exists. PRD-CHAT is `next`, AIREC is `later`, so AIREC depends on a chat event contract that is never defined; if "sustained" (the 7-day threshold) changes, AIREC's training data silently degrades. → **Blown apart because:** the AIREC team must invent an undocumented integration point; the story is not Independent per INVEST and cannot be estimated or delivered cleanly. → **Rewrite it properly:** add `depends_on: [PRD-CHAT]` and an AC pinning the contract, "Given a conversation reaches the 7-day sustained threshold, when the chat service emits the event, then AIREC's signal store receives {user_pair_id, match_id, event_type:'sustained_7d', timestamp} within 60s."

- **[major] PRD-PREMIUM-E1-S1:20**: the AC fires "when the transaction completes", presupposing a working payment transaction, while PRD-PREMIUM's Won't section explicitly defers payment-gateway integration this round. The AC tests a flow the spec simultaneously says will not be built. → **Blown apart because:** a tester cannot exercise it; the story gets stubbed (defeating the AC) or blocked on an out-of-scope dependency, and the contradiction surfaces mid-sprint. → **Rewrite it properly:** either bring payment-gateway integration into scope with its own story, or decouple, "Given the payment service emits a successful-payment event for user X / plan Y, when the entitlement service receives it, then the who-liked-me flag for X is true within 5s", testable with a stubbed event while the real gateway stays separate. `DEC-worthy` (scope ruling: AC precondition contradicts a Won't clause).

- **[minor] PRD-MATCH-E1-S2:19**: the 3/4 hard-criteria filter AC assumes a user preference-setting UI and stored preference profile, but no story builds the filter/preference UI. MATCH's Should mentions "basic filters" with no implementing story. → **Blown apart because:** with no user-side input, the filtering AC cannot be tested; the team hardcodes defaults (violating the AC) or blocks the story mid-sprint. Not Independent per INVEST. → **Rewrite it properly:** add a prerequisite story ("user sets age range, max distance, desired gender, serious-intent toggle; persisted server-side, returned in the profile API") and add `depends_on: [that story id]` to S2.
