### PSC-1 · CORRECTNESS · HIGH · `.claude/skills/product-spec-critique/scripts/critique_inherit.py:59`
- **Root cause:** the finding-fingerprint normalizer stripped a leading list **number**
  along with markdown markers, so two distinct findings citing same-prose ordered-list
  siblings (`3. xem số dư` vs `7. xem số dư`) on the same node+severity hashed identical
  → the index merged them → repeat-offense undercount (the exact class the fingerprint
  exists to remove, reintroduced for numbered lines).
- **Before:** an over-broad `^(?:#{1,6}|>+|[-*+]|\d+\.)\s+` marker regex normalized both
  siblings to `"xem số dư"` → one index row for two findings.
- **Fix:** `_MARKER_RE` drops only heading/quote/bullet markers; leading list numbers are
  kept as content, so numbered siblings stay discriminable. Identical *post-normalize*
  prose on one node+severity is still one finding — by design (same content = same finding).
- **After:**
  ```
  $ .claude/skills/.venv/bin/python3 -m pytest scripts/tests/test_critique_inherit.py \
      -k "numbered_siblings_distinct or overstrip" -q
  ..                                                                       [100%]
  ```
- **Note:** no DEC needed (no contradiction with an approved artifact — a correctness fix).

### LIB-1 · CORRECTNESS · MEDIUM · `.claude/skills/_shared/lib/run_evals.py:245`
- **Root cause:** the `--root` default walked `parents[3]` from `.claude/skills/_shared/lib/run_evals.py`,
  landing on `.claude/` not the repo root, so `{repo}`-relative exec paths resolved with a doubled
  `.claude` segment → every scenario FATAL'd before any checker ran (a false-RED, not a false-green).
- **Before:** `default=...parents[3]` → `{repo}` expanded to `<repo>/.claude` → `python -m pack` could not

  ```
  claude-pack: 10 pass · 0 fail · 0 skip(manual) · 5 unmapped   # exit 0; was FATAL before
  ```
- **Note:** caught during A3 wiring, before any CI relied on it; no by-design caveat.

### LIB-2 · CORRECTNESS · MED · `.claude/skills/_shared/lib/llm_eval.py`
- **Root cause:** A3's `run_evals.py` honestly reports every `llm_advisory` assertion as `SKIP(manual)` —
  correct (a script must not fake a judgment), but it left 35 advisory assertions (13 product-spec +
  22 critique) with *no runnable path at all*. The judgment gap was visible yet unexercisable.
- **Before:** advisory assertions could only ever be SKIP; there was no way to actually judge "tone matches
  level 4" / "no fabrication" / "floor holds" against a produced report without a human reading each.
  ```
  $ .claude/skills/.venv/bin/python3 .claude/skills/_shared/lib/run_evals.py --skill product-spec-critique
  product-spec-critique: 0 pass · 0 fail · 22 skip(manual) · 29 unmapped   # the 22 advisory: unrun, by design
  ```
- **Fix:** `llm_eval.py` — the advisory half of the gate. Judges each `llm_advisory` assertion against a
  **golden artifact fixture** (`eval/golden/<id>.md`) via an *injectable* LLM client. Verdicts: PASS / FAIL /
  MISSING (golden absent — loud, non-gating unless `--strict-missing`) / ERROR (unparseable judge — gating).
  Never fabricates: a missing golden never calls the judge, and a garbage reply is ERROR not PASS. The real
  client (stdlib `urllib`, no SDK) is built only when creds exist; `FakeLLMClient` drives all 15 tests with
  no network/key. `--gen` drafts missing goldens (DRAFT-marked, never overwrites) — the deferred path that
  needs an API key + human review.
- **After (real LLM run, 6 scenarios with grounded goldens — 5 mapped to committed example reports
  via the `golden` key, 1 generated by a subagent that read the real acme-shop spec):**
  ```
  $ .claude/skills/.venv/bin/python3 -m pytest .claude/skills/_shared/lib/__tests__/test_llm_eval.py -q
  16 passed
  $ .claude/skills/.venv/bin/python3 .claude/skills/_shared/lib/llm_eval.py --skill product-spec-critique
  product-spec-critique: 6 pass · 1 fail · 15 missing · 0 error   # real verdicts, not SKIP
  ```
  The 1 FAIL is a *true* signal: the level-8 example renders `mày/tao`, but scenario 7 specifies
  `critique_dialect=trung` → should render `mi/tau`; the judge caught the example does not cover that knob.
- **Note:** by design NOT in CI — the judge needs a key and its verdicts need review; local on-demand
  advisory gate, run when goldens are generated + human-approved. Closes the honest-but-unrun half A3 left.
  Two robustness fixes surfaced only by running it for real against a live gateway: (1) the stdlib client
  must send a real `user-agent` (a Cloudflare-fronted endpoint returns HTTP 403 / CF-1010 on urllib's
  default signature); (2) `parse_verdict` accepts a boolean/`yes`/`no` verdict, not only the `"PASS"`/`"FAIL"`
  string — a model returned `{"verdict": true}` and must not fall through to ERROR. Preferred golden source
  is a real example report (`golden` key) or a subagent that runs the skill; the urllib `--gen` is a rough
  ungrounded bootstrap only.

## Cycle 1 — 2026-06 (release-skill rename + bundle restructure)

### PACK-2 · CORRECTNESS · HIGH · `.claude/skills/release/scripts/release.py:26`
- **Root cause:** the new release-lifecycle engine resolves the repo root by parent-walking from its own path;
  the migration plan *guessed* `REPO=parents[3]`, which would land on `.claude/` (not the repo root) and make
  every `--release/--extract` read/write the WRONG `CHANGELOG.md` + `pack.manifest.yaml` — a release cut against
  a non-file, or worse a silent write to a stray path.

- **Fix:** `REPO = Path(__file__).resolve().parents[4]` (scripts → release → skills → .claude → repo), matching
  the file's real nesting (identical depth to human-analyzer's `_framework-shared/release.py`). Locked by a unit
  test that asserts `REPO/.claude/pack.manifest.yaml` and `REPO/CHANGELOG.md` both exist.
